Hamming Codes

              While using technology, one does not often take time and reflect on the discoveries that made everything possible, especially the particularly low-level concepts that are integral to almost all modern technologies. Error correction is certainly one of those concepts, being necessary for any field that works with data, which is to say every area of computer science. Hamming codes are one of the first examples of efficient error correction, and it is the one implemented in the accompanying technological demonstration.

Premise

              At first glance, error correction seems easily accomplished with a simple addition of redundant copies of data. While this may be almost effortless to implement, it is far from efficient or even feasible in a lot of cases due to constraints regarding the amount of data being transmitted/stored. Hamming codes are capable of error resilience up to 1 bit with the amount of non-information data only being approximately equal to the binary logarithm of the total amount of transmitted data.

Parity

              At the core of binary error correction is the concept of parity. Every bit string can be classified into either even or odd parity – meaning the number of 1-bits can either be even or odd. When a bit string is altered, the parity is also changed, meaning that it is possible to detect when a singular error has occurred, but not the location of said error. However, a singular parity check cannot detect an even number of distinct errors.

(7, 4)

        Using multiple parity checks, we can map out where the error has occurred in a 2-dimensional representation of the data. These parity checks happen at specific spots to determine both the row and column of the affected bit. Parity bits are added before transmision/storage such that the new parity of their given area is even. This way, an error is defined by odd parity.

        In this example, the positions reserved for the parity checks are 1, 2, and 4. Since this means 4 bits are reserved for information, this configuration is called (7, 4).

Notice that if any error is detected by the parity checks, the row and column of the affected bit can be deduced. For example, if errors are found in all three parity checks, then the affected position can be deduced to be 7.

        Additionally, notice that the first parity check involves all positions whose binary representation’s first bit is a 1. This rule holds for the second and third check as well, for the second and third bits respectively. In practice, this means that the parity checks effectively “spell” the position of the error in binary.

        In practice, this property is implemented through a bitwise XOR (addition mod 2) of all positions of 1-bits in the bit string. This works because XOR is essentially a parity check, which means that if there is an odd parity in a certain area, it is represented as a 1 in its respective bit in the position value.

In addition to the configuration above, one can also add an extra parity bit in the 0th position that records the parity of the entire bit string. This addition allows the detection of double-errors, but cannot locate them. This is due to how a double error appears invisible to the parity check of the entire bit string, but still appears as a single error to the other parity checks. This contradiction is exclusive to an error count of two or more, although it’s not guaranteed for errors counts higher than two. The name of this new configuration is (8,4)

Scaling

        Configurations of Hamming Codes are scalable to any size such that the total amount of data (including parity bits) is a power of two. Additionally, the efficiency of the configuration increases with size. A downside, however, is that since Hamming Codes are only resilient to errors of up to one bit, multiple errors are more likely to occur in a bigger configuration as opposed to multiple smaller configurations.

Implementation

        

        In practice, errors are likely to occur in bursts. This means that even when the data is broken up into smaller “chunks” of Hamming Codes, it is still likely that a whole chunk could be irreparable due to two or more errors occurring. To prevent this, the practice of interlacing the chunks is used such that a burst error only affects one bit per affected chunk.
